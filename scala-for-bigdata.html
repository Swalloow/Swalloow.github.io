빅데이터 처리에 Scala가 필요한 이유

17 Mar 2017 on Develop

​ ​

Spark를 시작하기 이전에 Scala 라는 언어에 대해 정리해보려고 합니다. 스칼라 개발환경은 Scala 2.12.1 이며, IDE는 IntelliJ IDEA 를 사용하였습니다.

​ ​

Scala에 대해 알아보자

Scalable Language

스칼라는 모든 것들이 일관성있게 구현되도록 설계되었습니다. 이를 통해 얻을 수 있는 장점은 “적은 양의 코드로 방대한 규모의 시스템을 작성할 수 있다” 는 것입니다. int형 데이터를 예를 들면, 자바에서는 Primitive type인 int와 Integer class가 존재합니다. 만약 범위를 벗어나는 큰 수를 다루어야 한다면 BigInteger를 사용해야 합니다. 스칼라에서는 같은 숫자인데 다른 타입으로 연산할 필요가 없습니다. 스칼라는 항상 간결하고 일관성 있는 표현을 지향하기 때문입니다.

이번에는 비교연산자를 예로 들어보겠습니다. 자바에서는 ‘==’ 와 같은 비교연산자를 제공합니다. 하지만 비교연산자는 주소값을 비교하기 때문에 String과 같은 객체를 비교할 때는 ‘equal()’ 메서드를 사용해서 비교해야 했습니다. 이 또한 스칼라의 Scalable과 거리가 멉니다. 스칼라에서는 모든 것이 Object이기 때문에 ‘==’ 로 모든 비교가 가능합니다.

​ ​

Functional Language

스칼라에서는 함수가 first object 입니다. 함수를 마치 하나의 값으로 취급하며 이를 변수 또는 파라메터로 넘길 수 있습니다.

스칼라의 val 변수는 한번 지정된 값을 바꾸지 않습니다. 이러한 변수를 Immutable variable 이라고 부릅니다. 예를 들면 아래와 같습니다.

위의 예제를 보면, msg 변수에 문자열을 할당했지만 어디에도 String 이라는 단어는 없습니다. 스칼라는 알아서 타입을 추론하여 지정해주기 때문입니다. 따라서, val 변수에 재할당을 시도하면 reassignment to val 이라는 오류가 발생하게 됩니다. 하지만, 필요한 경우에 기존과 같이 재할당이 가능한 val 변수를 지원합니다.

함수형 프로그래밍은 의도하지 않은 동작이 없습니다. (Side Effect) 이를 통해 버그를 줄여준다. 모든 것을 함수로 해결하면 side effect가 발생할 일이 없고 한번 검증된 함수는 신뢰할 수 있기 때문에 버그 걱정이 없다. 데이터가 share되는 경우 immutable 변수가 문제를 단순화 해주고 병렬처리에 강함.

빅데이터 분석에는 정적 타입이 더 좋다? 컴파일시 타입 체킹, 성능이 좋다. 잘못된 타입을 넣었을 때, 실행을 시켜봐야 에러를 알 수 있음 (단점) 스칼라는 input 타입을 보고 함수나 출력 값의 타입을 추론해준다. 이를 통해 코드를 깔끔하게 유지해준다 (implicit conversion)
